# HWPX 정부지원사업 사업계획서 스킬

정부지원사업 사업계획서를 **공고문 분석 → 초안 작성 → 휴먼라이징 → DOCX 생성 + HWPX 채우기 → 검토 → 제출**까지 end-to-end로 처리하는 스킬.

## 산출물 형식: DOCX(메인) + HWPX(보조)

**DOCX가 메인 산출물**이다. python-docx로 양식 구조에 맞춘 DOCX를 직접 생성하며, 테이블·본문 서술·시각 자료를 모두 포함한다. 제출 시 HWP/PDF 변환이 필요하면 한글에서 DOCX를 열어 HWP로 저장하거나, 워드에서 PDF로 변환한다.

**HWPX는 보조 산출물**이다. hwpx_handler.py로 원본 양식의 테이블 셀에 데이터를 채운다. 단, HWPX 직접 편집에는 한계가 있다(아래 "HWPX 편집 시 알려진 한계" 참조). HWPX가 열리지 않을 경우 DOCX를 메인으로 사용한다.

## 언제 이 스킬을 사용하는가

- 사용자가 정부지원사업 사업계획서를 작성하려 할 때
- 모집공고문(.hwpx/.pdf)과 작성서식(.hwpx)이 주어졌을 때
- HWPX 파일의 표를 읽거나 수정할 때
- 사업계획서에 시각 자료를 넣을 때

## 실행 모드

사용자가 "사업계획서 작성해줘" 등으로 요청하면, 먼저 모드를 물어본다:

### 모드 A: 협업 모드 (기본값, 추천)
AI가 초안을 작성하되, **핵심 의사결정 포인트에서 사용자에게 질문**한다.
질문하는 시점:
1. 사업명 확정 (후보 3개 제시 → 택1)
2. 사업 내용의 핵심 방향 (어떤 기능을 고도화할지)
3. 예산 배분 비율 (초안 제시 → 조정)
4. 정량적 목표치 (초안 제시 → 조정)
5. 최종 확인 (HWPX 저장 전)

### 모드 B: 자동 모드
AI가 모든 결정을 내려 **한 번에 완성본을 생성**한다.
사용자는 결과물만 검토/수정하면 된다.
- 사업명, 예산, 목표 등을 AI가 최적으로 결정
- 완성 후 "수정할 부분 있으세요?" 한 번만 질문

사용자에게 이렇게 물어본다:
> **어떤 모드로 진행할까요?**
> - **A) 협업 모드** — 중요한 결정은 함께 (추천)
> - **B) 자동 모드** — 전부 맡기고 결과만 확인

## Phase 0: 파일 자동 탐색 (스킬 시작 시 항상 실행)

스킬이 시작되면 **파일 경로를 묻기 전에** 현재 작업 디렉토리를 먼저 탐색한다.

```bash
# 현재 디렉토리 및 하위 폴더에서 관련 파일 자동 탐색
find . -maxdepth 3 \( \
  -name "*.hwpx" -o -name "*.hwp" \
  -o -name "*.pdf" \
  -o -name "*.docx" -o -name "*.doc" \
  -o -name "*.pptx" -o -name "*.ppt" \
  -o -name "*.xlsx" -o -name "*.xls" \
  -o -name "*.txt" -o -name "*.md" \
  -o -name "*.key" -o -name "*.pages" -o -name "*.numbers" \
\) 2>/dev/null | sort
```

탐색 결과를 아래 용도로 분류하여 사용자에게 제시한다:

| 용도 | 파일 패턴 예시 |
|------|-------------|
| 모집공고문 | `*공고*`, `*모집*`, `*지원사업*` |
| 작성서식(양식) | `*서식*`, `*양식*`, `*template*`, `*신청서*` |
| 회사 참고자료 | `*소개서*`, `*재무*`, `*IR*`, `*사업자*`, `*계획서*` |
| 기타 | 위에 해당하지 않는 나머지 |

**분류가 불명확한 파일만** 사용자에게 용도를 확인한다.  
파일이 전혀 없으면 "공고문과 작성서식 파일을 이 폴더에 넣어주세요" 안내 후 대기.

## Phase 1: 공고문 + 양식 분석 (자동)

### 1-1. 공고문 분석
Phase 0에서 탐색한 공고문 파일(HWPX/PDF)에서 텍스트를 추출하고 구조화한다.

추출 항목:
- 사업 개요 (사업명, 목적, 기간, 지원금액, 모집규모)
- 지원 자격 요건 (기본요건, 우대요건, 결격사유)
- **평가 기준** (항목별 배점, 세부 평가내용) ← 가장 중요
- 제출 서류 목록 + 형식
- 주요 일정 (마감일, **중간평가일**, 최종평가일, 협약일)
- 예산 편성 규칙 (비목, 세목, 사용가능 범위, **비목별 지출 한도 비율**)
- **주관기관 지역** (지역성 강조 포인트 파악)
- **주관기관 유형** (심사역 프로파일 추론용)

### 1-1-1. 지원 유형 분류 (전략 방향 결정)

공고문 분석 직후, 아래 유형 중 하나로 분류한다. **유형에 따라 전체 전략이 달라진다.**

| 유형 | 핵심 판단 신호 | 전략 가중치 |
|------|-------------|-----------|
| **예비창업패키지** | "예비창업자", 매출 0 허용 | 아이디어 신선도·창업 의지 최대화. 기존 매출 강조 금지 |
| **창업도약패키지** | "3년 이내", 스케일업 강조 | 시장 검증 증거(계약서·매출)·스케일업 시나리오 중심 |
| **TIPS/딥테크** | "기술성", 투자사 연계 | 기술 차별성·논문·특허·투자확약서 중심 |
| **지역 R&D/사업화** | 지자체 주관, 지역 명시 | 지역 기여·지역 파트너·지역 고용 극대화 |
| **수출바우처/글로벌** | "수출", "해외진출" | 수출 실적 또는 해외 파이프라인 구체성 중심 |
| **일반 사업화** | 위에 해당 없음 | 사업성·실행가능성·팀 역량 균형 |

분류 결과를 `_사업계획서_준비자료.md`에 기록하고, 이후 모든 Phase에서 이 유형을 기준으로 강조 포인트를 조정한다.

### 1-1-2. 주관기관 프로파일 → 심사역 성향 추론

주관기관명과 유형을 분석하여 심사역 성향을 추론하고, 작성 톤을 조정한다.

| 주관기관 유형 | 심사역 성향 추론 | 작성 톤 조정 |
|------------|--------------|-----------|
| 창진원·중기부 직접 | 창업 멘토·전문 심사역 多 | 숫자·스케일업 논리 강조, 전문용어 허용 |
| 테크노파크·진흥원 | 현장 경험자 + 교수 혼합 | 기술 실현 가능성 + 사업성 균형 |
| 지자체 직접 주관 | 지역 공무원 비율 높음 | 전문용어 최소화, 지역 기여 극대화 |
| TIPS 운용사(VC) | 투자자 관점 | Unit Economics·시장 점유율·Exit 시나리오 |
| 대기업·공기업 | 업종 전문가 | 해당 산업 용어·레퍼런스 적극 활용 |

추론 결과를 `_사업계획서_준비자료.md`에 기록한다.

### 1-2. 양식 분석
작성서식 HWPX의 테이블 구조를 파악한다.
```bash
PYTHON=~/.claude/skills/k-proposal/.venv/bin/python3
HANDLER="$PYTHON ~/.claude/skills/k-proposal/hwpx_handler.py"

# 전체 구조 (처음 3행)
$HANDLER analyze "양식.hwpx"

# 특정 테이블 전체 행 (✎=빈셀/입력가능, [NxM]=셀 병합)
$HANDLER analyze "양식.hwpx" -t 1 -a

# 테이블 일괄 읽기 (1회 호출로 전체 셀 구조 + 빈셀 확인) ← 권장
$HANDLER read-table "양식.hwpx" -t 1
$HANDLER read-table "양식.hwpx" -t 3 --json  # JSON 형식
```

산출물: 테이블 인덱스, 행/열, 셀 매핑(빈셀 ✎ 표시), 병합 정보, 작성요령 테이블 위치

### 1-3. 준비자료 생성
분석 결과를 `_사업계획서_준비자료.md`로 저장한다.
내용: 공고 요약, 평가기준표, **지원 유형**, **주관기관 프로파일**, 예산규칙(비목별 한도 포함), 양식 테이블맵, **광탈 패턴 목록**, **지역성 포인트**

## Phase 3: 초안 작성

### 3-0. 광탈 패턴 사전 배제 (작성 전 체크)

초안 작성을 시작하기 전에 아래 광탈 패턴을 **절대 사용하지 않도록** 내부 규칙으로 설정한다.

**언어 광탈 패턴 (쓰면 감점):**
- "세계 최고 수준의 AI 기술" — 근거 없는 최상급 표현
- "국내 유일" / "독보적" — 검증 불가 주장
- "혁신적인" / "차세대" / "게임체인저" — 추상적 수식어
- "본 사업은 ~을 목적으로 하며" — AI 작성 흔적 1순위
- "이를 통해 ~의 효과를 기대할 수 있습니다" — AI 작성 흔적 2순위
- "~할 것으로 기대됩니다" — 막연한 미래 추정
- "4차 산업혁명" / "디지털 전환" — 남발된 유행어
- "글로벌 시장 진출" (6개월 안에 실현 불가한 목표)

**구조 광탈 패턴 (구성 시 주의):**
- 10페이지 중 7페이지 이상이 기술 설명 → "사업화 의지 부족"
- 정량적 목표가 0개이거나 너무 낮음 → "성의 없어 보임"
- 정량적 목표가 비현실적으로 높음 → "신뢰 불가"
- 예산 산출근거가 "1식" 단위 → 감점 확정
- 경쟁사 분석 없음 → 시장 이해도 부족
- 측정 불가능한 목표 (증빙 없이 달성 불가)

### 3-1. 평가기준 매칭 전략
각 평가항목(배점)에 회사의 강점/약점을 매칭하고, 섹션별 작성 전략을 수립한다.

### 3-2. 핵심 3문 답변 (작성 전 필수 확인)

협업 모드/자동 모드 모두에서 아래 3개 질문에 명확한 답을 먼저 정의한다. 이 답이 사업계획서 전체의 논리 축이 된다.

**질문 1: Why Now — 왜 지금인가?**
> 시장/기술/규제 환경에서 지금 이 시점이 특별한 이유는?
> (시장 성장률 데이터, 정책 변화, 경쟁 구도 변화 등 근거 포함)

**질문 2: Why Us — 왜 이 팀인가?**
> 경쟁사 대비 우리 팀이 이 문제를 더 잘 해결할 수 있는 구체적 근거는?
> (특허, 핵심인력 경력, 기존 고객사, 보유 데이터 등)

**질문 3: Why 지원 필요 — 지원 없이 왜 안 되는가?**
> 이 자금이 없으면 사업이 어떻게 지연/불가해지는가?
> ※ 심사역이 가장 중요하게 보는 질문. "이 회사가 자력으로 할 수 있으면 지원 불필요"

아래 세 가지를 회사 자료(재무제표)에서 직접 계산하여 수치로 제시한다. **AI가 임의로 숫자를 만들지 않는다.**

**① 자력 불가 증명 (현금 잔고 vs. 개발 비용)**
```
현재 현금 잔고:        X원  (재무제표 유동자산 기준)
이 사업 개발 비용:     Y원  (공고 지원금액 기준)
자체 조달 가능 금액:   Z원  (현금 × 가용 비율 추정)
갭:                   Y - Z = 부족분  → "자력 불가" 근거
```

**② BEP 시뮬레이션 (지원금 있을 때 vs. 없을 때)**
```
지원금 있을 경우:  M개월 후 손익분기점 도달 (Unit Economics 기반)
지원금 없을 경우:  개발 지연 N개월 → BEP N+개월 후로 밀림
```

**③ 런웨이 계산 (지원 없으면 언제 자금 소진)**
```
월 고정비: X원  (인건비 + 운영비)
현금 잔고: Y원
런웨이:    Y ÷ X = N개월 → 지원 없으면 N개월 후 자금 소진
```

재무제표 데이터가 없으면 사용자에게 정확한 수치를 요청한다. 추정치를 사용할 경우 반드시 "[추정]" 표시.

### 3-3. 의사결정 (협업 모드에서만)
아래 순서로 사용자에게 질문한다:

**질문 1: 사업명**
> 아래 후보 중 선택하거나 수정해 주세요:
> A) [후보1]

### 3-4. 섹션별 초안 작성
마크다운으로 초안을 작성하고, 사용자에게 보여준다.

**작성 순서:**
1. **요약표 초안** — 먼저 뼈대를 잡는다 (심사역이 제일 먼저 읽는 곳)
   - 각 셀에 한 줄씩 키워드만 채워 넣는다
   - 본문 작성 후 7단계에서 최종 다듬는다
2. **사업 내용** — 사업성(30점) + 수행역량(30점) 대응
   - Why Now (시장 타이밍 데이터 포함)
   - Why Us (아래 경쟁사 분석 구조 적용)
   - Why 지원 필요 (3-2의 BEP + 런웨이 수치 반영)
   - Unit Economics (LTV, CAC, 목표 마진 등)
3. **지원 동기/시급성** — 참여의지(20점) 대응
4. **기대효과** — 기대효과(20점) 대응
   - 모든 기대효과는 **계량화된 수치 필수** (예: "매출 00억원", "생산성 향상 00% → 00%", "일자리 창출 00명")
   - 매출 목표는 사업 규모에 맞게 **공격적으로 설정** — 특히 글로벌 진출 사업은 국내 매출만으로는 부족하며, 해외 매출 포함한 성장 시나리오 제시
   - "기여할 것으로 기대됨" 같은 막연한 표현 금지 → 반드시 구체적 수치와 달성 시점 명시
5. **인력/파트너/기술** — 수행역량 보강
6. **목표/일정/예산** — 수행역량 보강
7. **지역성 섹션** — 지역 기여도 및 지역 생태계 연계
8. **요약표 최종 다듬기** — 본문 전체 반영 후 첫인상 최적화

### 3-5. 팩트체크 & 리서치 보강

초안 작성 후, 아래를 자동으로 수행한다:

#### (a) 팩트체크
- 재무 수치가 제출 자료(재무제표)와 일치하는지 확인
- 매출 증빙 금액과 기재 금액이 일치하는지 확인
- 설립일, 주소 등 기본정보가 사업자등록증과 일치하는지 확인
- 기재한 수상/선정 이력이 참고자료에서 확인되는지 체크

#### (b) 시장 리서치 (WebSearch/WebFetch 사용)

**🚫 핵심 원칙: 검증 불가 데이터는 사실처럼 쓰지 않는다.**

##### 단계 1: 검색
- 타겟 시장 규모 데이터 검색 (예: "ESG 시장 규모 2026", "임팩트 측정 시장 성장률")
- 경쟁 서비스 현황 검색 (Why Us 근거 강화)
- 관련 정책/트렌드 뉴스 검색 (Why Now 근거 강화)

##### 단계 2: 검증
모든 시장 데이터(시장 규모, 성장률, 점유율, 통계 수치)는 아래 기준을 충족해야 한다:
- **출처 3요소 필수**: 보고서명(또는 기관명) + 발행연도 + URL(또는 접근경로)
  - 예: "한국임팩트투자네트워크, 2025 임팩트금융 현황 보고서, https://example.com/report"
- WebSearch로 찾은 수치는 **WebFetch로 원문 페이지를 열어 수치가 실제 존재하는지 재확인**
- 출처 3요소를 채울 수 없는 데이터는 **절대 사실로 기재하지 않는다**

##### 단계 3: 분류 후 기재
검증 결과에 따라 세 등급으로 분류한다:

| 등급 | 조건 | 기재 방식 |
|---|---|---|
| ✅ 검증됨 | 출처 3요소 확보 + 원문 확인 완료 | 수치 + 출처 각주 |
| ⚠️ 추정치 | 검증된 상위 데이터에서 산출한 추정 | "~로 추정됨" 명시 + 산출 근거 기재 |
| ❓ 확인 필요 | 출처 확보 실패 또는 원문 불일치 | "[확인 필요]" 태그 + 사용자에게 보완 요청 |

**금지 사항**:
- WebSearch 결과 없이 숫자를 생성하는 행위 (예: "약 X조 원 규모로 추정된다")
- 검색 결과의 숫자를 다른 맥락에 전용하는 행위
- 출처 없이 "~에 따르면", "업계에 따르면" 등의 표현 사용

**⚠️ 시장 데이터 적합성 검증 (필수)**:
- 인용하는 시장 데이터가 신청 제품/서비스의 **직접 타겟 시장**인지 반드시 검증
- 심사위원이 "이 시장 데이터가 왜 이 제품과 관련 있는가?"라고 물었을 때 한 문장으로 답할 수 있어야 함
- 직접 시장 데이터가 없으면 상위 시장 데이터 + 연결 논리를 명시적으로 서술
- 예시:
  - (X) ESG 보고 소프트웨어 시장 → 임팩트 측정 서비스에 간접 관련 (시장 카테고리 불일치)
  - (O) IMM(임팩트 측정·관리) 시장 → 임팩트 측정 서비스에 직접 관련

#### (c) 논리 보강
- Why Now / Why Us / Why 지원 필요 세 논리가 모두 근거를 갖췄는지 점검
- 숫자 근거가 없는 주장에 데이터 추가
- "~할 것이다" 같은 막연한 표현을 구체적 수치로 보강
- 평가항목과 매칭되지 않는 빈 영역 식별 → 보완
- **광탈 패턴 최종 점검** — Phase 3-0의 목록과 대조

산출물: 초안 마크다운 + 팩트체크 결과 + 리서치 근거

### 3-7. 시각 자료 생성

### HWPX 채우기 워크플로우 (검증된 4단계)

⚠️ remove-guides, insert-text는 파일 손상 위험이 높다. 대신 아래 워크플로우를 사용한다.

```
Step 1. 원본 복사 + fill 명령으로 테이블 셀 채우기
Step 2. lxml 직접 조작으로 개조식 단락(ㅇ/-/◦) 채우기 (텍스트 패턴 매칭)
Step 3. 후처리: 원본 대비 diff → linesegarray 빈 태그 + charPrIDRef 변경 + 볼드+밑줄 강조 + 네임스페이스 정규화
Step 4. 이미지 삽입 (insert_image.py)
```

#### Step 1: 테이블 셀 채우기
```bash
cp "양식.hwpx" "제출용.hwpx"
$HANDLER fill "제출용.hwpx" "제출용.hwpx" --data fill_step1.json  # 기업정보
$HANDLER fill "제출용.hwpx" "제출용.hwpx" --data fill_step2.json  # 요약표, 인력, 파트너 등
```

#### Step 2: 개조식 단락 채우기 (lxml 직접 조작)
hwpx_handler의 remove-guides/insert-text 대신 lxml로 직접 처리한다.
- 앞 단락의 고유 텍스트(예: "주력 사업 현황")를 키로 사용하여 정확한 위치 특정
- `<hp:t>` 텍스트를 직접 교체하거나, `<hp:t>`가 없으면 `<hp:run>` 안에 생성
- **인덱스 기반이 아닌 텍스트 패턴 매칭** 사용 (fill 후 인덱스 변경 위험 회피)

```python
# 패턴 매칭 예시
SECTIONS = [
    ('주력 사업 현황', [  # 앞 단락의 고유 텍스트
        ("   ㅇ 첫 번째 내용", "     - 상세 설명"),  # (ㅇ, -)
        ("   ㅇ 두 번째 내용", "     - 상세 설명"),
    ]),
    ...
]
```

#### Step 3: 후처리 (필수!)
```python
# 원본과 수정본의 모든 <hp:p> 텍스트를 비교
for i, (mod_p, mod_text) in enumerate(mod_texts):
    if mod_text != orig_text and mod_text.strip():
        # 1) linesegarray → 빈 태그 (글자 겹침 방지)
        for lsa in mod_p.findall(f'{HP}linesegarray'):
            mod_p.remove(lsa)
        etree.SubElement(mod_p, f'{HP}linesegarray')
        # 2) charPrIDRef → 79 (검정 맑은고딕, 파란색 가이드 스타일 제거)
        for run in mod_p.iter(f'{HP}run'):
            run.set('charPrIDRef', '79')
        # 3) 키워드 볼드+밑줄 (run 분할 + 강조 charPrIDRef 적용)
```

**볼드+밑줄 구현 방법:**
1. header.xml에서 기존 검정 맑은고딕 charPr(예: ID=79)를 복사
2. `<bold/>` + `<underline type="BOTTOM" shape="SOLID" color="#000000"/>` 추가
3. 새 charPr ID로 등록 (charProperties의 itemCnt 업데이트)
4. 강조할 키워드가 포함된 run을 분할: 일반 텍스트(charPr=79) + 강조 텍스트(charPr=새ID)

## HWPX 파일 구조

> 상세 구조, 네임스페이스 매핑, 단위 변환, XML 요소 참조는 → **`references/xml-internals.md`** 참조

HWPX는 ZIP + XML 구조. 핵심 파일:
- `Contents/section0.xml` — 본문, 표, 이미지, 텍스트
- `Contents/header.xml` — 스타일(charPr, paraPr), 폰트
- `Contents/content.hpf` — OPF manifest (이미지 등록)
- `BinData/` — 이미지 파일
- `mimetype` — 반드시 첫 엔트리, ZIP_STORED

핵심 XML 경로:
- **테이블**: `hp:tbl` > `hp:tr` > `hp:tc` > `hp:subList` > `hp:p` > `hp:run` > `hp:t`
- **이미지**: `hp:p` > `hp:run` > `hp:pic` > `hc:img[binaryItemIDRef]`
- **텍스트**: `hp:p` > `hp:run[charPrIDRef]` > `hp:t`

## 주의사항

### 공통 주의사항
1. **분량 제한**: 공고문에 명시된 페이지 이내로 작성
2. **팩트체크 필수**: 재무 수치, 이력, 날짜는 반드시 원본 자료와 대조
3. **AI 흔적 제거 타이밍**: 초안 완성 직후, 문서 생성 전에 반드시 수행
4. **증빙 가능성**: 정량 목표는 반드시 증빙 가능한 형태로 설정
5. **예산 산출근거**: "1식" 단위 금지, 공수 기반으로만 작성

### DOCX 주의사항
6. **양식 구조 파악 필수**: HWPX analyze로 원본 양식의 섹션 구조(Ⅰ~Ⅵ)와 개조식 서술 영역(ㅇ / - / ◦)을 먼저 파악한 뒤 DOCX 생성
7. **시각 자료 삽입**: matplotlib로 생성한 차트를 doc.add_picture()로 삽입. 이미지는 `_시각자료/` 폴더에 별도 보관
8. **폰트**: macOS에서는 AppleGothic, Windows에서는 맑은 고딕 사용. DOCX 스타일에서 font.name 지정
9. **제출 변환**: DOCX → HWP가 필요하면 한글에서 DOCX를 열어 HWP로 저장. DOCX → PDF는 워드에서 변환

### HWPX 주의사항
10. **스타일 보존**: 셀 텍스트 쓸 때 기존 `charPrIDRef` 유지 (preserve_style=true)
11. **셀 병합 주의**: read-table의 `[NxM]` 표시로 병합 셀 확인 후 올바른 col 인덱스 사용
12. **fill 전 반드시 validate**: `fill --validate`로 좌표 오류를 사전 확인 (재시도 방지)
13. **행 추가 시 template-row**: 합계행이 아닌 데이터 행을 복제하도록 `--template-row` 지정 권장

### HWPX 편집 시 알려진 한계 (⚠️ 중요)

> 상세 원인·해결책 매트릭스는 → **`references/xml-internals.md`** "알려진 한계 및 해결책" 참조

주요 문제와 대응:

| 문제 | 대응 |
|------|------|
| 네임스페이스 프리픽스 깨짐 (ns0:/ns1:) | `fix_namespaces.py` 후처리 (필수) |
| 글자 겹침 | diff 기반 linesegarray 빈 태그 교체 |
| 파란색 글자 | charPrIDRef → 79(검정) 변경 |
| XML 선언 따옴표 | 큰따옴표 정규표현식 치환 |
| ZIP 메타데이터 변경 | 원본 ZIP 정보 복원 |
| 네임스페이스 선언 누락 (16개 필수) | register_namespace + fix_namespaces |

**대응 전략**:
- HWPX fill 후 반드시 **postprocess_hwpx.py 실행** (diff → lineseg + charPr + 볼드 + 네임스페이스 정규화)
- **remove-guides, insert-text는 파일 손상 위험** → 개조식 단락은 lxml 직접 조작(텍스트 패턴 매칭)으로 채움
- HWPX가 열리지 않으면 DOCX를 메인 산출물로 사용
- HWPX 편집 후에는 반드시 한글에서 열리는지 확인한 뒤 다음 단계 진행
