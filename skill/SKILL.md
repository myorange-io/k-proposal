# HWPX 정부지원사업 사업계획서 스킬

정부지원사업 사업계획서를 **공고문 분석 → 초안 작성 → 휴먼라이징 → DOCX 생성 + HWPX 채우기 → 검토 → 제출**까지 end-to-end로 처리하는 스킬.

## 산출물 형식: HWPX(메인) + DOCX(선택)

**HWPX가 메인 산출물**이다. 원본 양식의 구조를 그대로 유지하며 테이블 셀·개조식 단락·시각 자료를 채운다.

**DOCX는 선택 산출물**이다. 사용자가 명시적으로 요청할 때만 생성한다. 제출 기관이 DOCX를 요구하거나 HWP 변환이 필요한 경우 등.

## 언제 이 스킬을 사용하는가

- 사용자가 정부지원사업 사업계획서를 작성하려 할 때
- 모집공고문(.hwpx/.pdf)과 작성서식(.hwpx)이 주어졌을 때
- HWPX 파일의 표를 읽거나 수정할 때
- 사업계획서에 시각 자료를 넣을 때

## 실행 모드

사용자가 "사업계획서 작성해줘" 등으로 요청하면, 먼저 모드를 물어본다:

### 모드 A: 협업 모드 (기본값, 추천)
AI가 초안을 작성하되, **핵심 의사결정 포인트에서 사용자에게 질문**한다.
질문하는 시점:
1. 사업명 확정 (후보 3개 제시 → 택1)
2. 사업 내용의 핵심 방향 (어떤 기능을 고도화할지)
3. 예산 배분 비율 (초안 제시 → 조정)
4. 정량적 목표치 (초안 제시 → 조정)
5. 최종 확인 (HWPX 저장 전)

### 모드 B: 자동 모드
AI가 모든 결정을 내려 **한 번에 완성본을 생성**한다.
사용자는 결과물만 검토/수정하면 된다.
- 사업명, 예산, 목표 등을 AI가 최적으로 결정
- 완성 후 "수정할 부분 있으세요?" 한 번만 질문

사용자에게 이렇게 물어본다:
> **어떤 모드로 진행할까요?**
> - **A) 협업 모드** — 중요한 결정은 함께 (추천)
> - **B) 자동 모드** — 전부 맡기고 결과만 확인

## Phase 0: 파일 자동 탐색 (스킬 시작 시 항상 실행)

스킬이 시작되면 **파일 경로를 묻기 전에** 현재 작업 디렉토리를 먼저 탐색한다.

```bash
# 현재 디렉토리 및 하위 폴더에서 관련 파일 자동 탐색
find . -maxdepth 3 \( \
  -name "*.hwpx" -o -name "*.hwp" \
  -o -name "*.pdf" \
  -o -name "*.docx" -o -name "*.doc" \
  -o -name "*.pptx" -o -name "*.ppt" \
  -o -name "*.xlsx" -o -name "*.xls" \
  -o -name "*.txt" -o -name "*.md" \
  -o -name "*.key" -o -name "*.pages" -o -name "*.numbers" \
\) 2>/dev/null | sort
```

탐색 결과를 아래 용도로 분류하여 사용자에게 제시한다:

| 용도 | 파일 패턴 예시 |
|------|-------------|
| 모집공고문 | `*공고*`, `*모집*`, `*지원사업*` |
| 작성서식(양식) | `*서식*`, `*양식*`, `*template*`, `*신청서*` |
| 회사 참고자료 | `*소개서*`, `*재무*`, `*IR*`, `*사업자*`, `*계획서*` |
| 기타 | 위에 해당하지 않는 나머지 |

**분류가 불명확한 파일만** 사용자에게 용도를 확인한다.  
파일이 전혀 없으면 "공고문과 작성서식 파일을 이 폴더에 넣어주세요" 안내 후 대기.

> **`.hwp` (구버전 한글) 파일을 발견한 경우 — 2가지 방법:**
>
> **방법 1 (자동, kordoc MCP 서버 설치 시):**
> kordoc MCP가 설치되어 있으면 `.hwp` 파일을 직접 파싱하여 마크다운으로 변환할 수 있다.
> ```bash
> # kordoc MCP의 parse_document 도구로 .hwp 읽기
> # → 공고문 분석/참고자료 분석에 활용 가능 (쓰기는 HWPX만 지원)
> ```
> 단, `.hwp`로의 **쓰기/생성은 불가**. 최종 산출물은 반드시 HWPX로 생성한다.
>
> **방법 2 (수동 변환):**
> kordoc MCP가 없거나 HWPX 양식이 필요한 경우:
> 한글에서 해당 파일을 열고 **파일 → 다른 이름으로 저장 → 파일 형식: HWPX**로 저장한 뒤 다시 알려주세요.

## Phase 1: 공고문 + 양식 분석 (자동)

### 1-1. 공고문 분석
Phase 0에서 탐색한 공고문 파일(HWPX/HWP/PDF)에서 텍스트를 추출하고 구조화한다.

**kordoc MCP 활용 (설치 시 자동 적용):**
- PDF 공고문: `parse_document`로 마크다운 변환 → 테이블 자동 추출, 머리글/바닥글 필터링
- HWP 공고문: `parse_document`로 레거시 HWP5 바이너리 직접 파싱 (수동 변환 불필요)
- `parse_form`으로 양식 필드 label-value 자동 인식 → 수동 매핑 작업 대폭 감소
- `parse_table`로 특정 테이블만 정밀 추출 (예산표, 평가기준표)

**텍스트 정제:**
추출된 텍스트에 `text_sanitizer.py`의 정제 함수를 적용한다:
- HWP 도형/OLE 자동생성 대체텍스트 제거 ("사각형입니다" 등 26종)
- 균등배분 공백 복원 ("현 장 대 응" → "현장대응")
- 연속 공백/탭 정리

추출 항목:
- 사업 개요 (사업명, 목적, 기간, 지원금액, 모집규모)
- 지원 자격 요건 (기본요건, 우대요건, 결격사유)
- **평가 기준** (항목별 배점, 세부 평가내용) ← 가장 중요
- 제출 서류 목록 + 형식
- 주요 일정 (마감일, **중간평가일**, 최종평가일, 협약일)
- 예산 편성 규칙 (비목, 세목, 사용가능 범위, **비목별 지출 한도 비율**)
- **주관기관 지역** (지역성 강조 포인트 파악)
- **주관기관 유형** (심사역 프로파일 추론용)

### 1-1-1. 지원 유형 분류 (전략 방향 결정)

공고문 분석 직후, 아래 유형 중 하나로 분류한다. **유형에 따라 전체 전략이 달라진다.**

| 유형 | 핵심 판단 신호 | 전략 가중치 |
|------|-------------|-----------|
| **예비창업패키지** | "예비창업자", 매출 0 허용 | 아이디어 신선도·창업 의지 최대화. 기존 매출 강조 금지 |
| **창업도약패키지** | "3년 이내", 스케일업 강조 | 시장 검증 증거(계약서·매출)·스케일업 시나리오 중심 |
| **TIPS/딥테크** | "기술성", 투자사 연계 | 기술 차별성·논문·특허·투자확약서 중심 |
| **지역 R&D/사업화** | 지자체 주관, 지역 명시 | 지역 기여·지역 파트너·지역 고용 극대화 |
| **수출바우처/글로벌** | "수출", "해외진출" | 수출 실적 또는 해외 파이프라인 구체성 중심 |
| **일반 사업화** | 위에 해당 없음 | 사업성·실행가능성·팀 역량 균형 |

분류 결과를 `_사업계획서_준비자료.md`에 기록하고, 이후 모든 Phase에서 이 유형을 기준으로 강조 포인트를 조정한다.

### 1-1-2. 주관기관 프로파일 → 심사역 성향 추론

주관기관명과 유형을 분석하여 심사역 성향을 추론하고, 작성 톤을 조정한다.

| 주관기관 유형 | 심사역 성향 추론 | 작성 톤 조정 |
|------------|--------------|-----------|

| 테크노파크·진흥원 | 현장 경험자 + 교수 혼합 | 기술 실현 가능성 + 사업성 균형 |

### 1-2. 양식 분석
작성서식 HWPX의 테이블 구조를 파악한다.
```bash
PYTHON=~/.claude/skills/k-proposal/.venv/bin/python3
HANDLER="$PYTHON ~/.claude/skills/k-proposal/hwpx_handler.py"

# 전체 구조 (처음 3행)
$HANDLER analyze "양식.hwpx"

# 특정 테이블 전체 행 (✎=빈셀/입력가능, [NxM]=셀 병합)
$HANDLER analyze "양식.hwpx" -t 1 -a

# 테이블 일괄 읽기 (1회 호출로 전체 셀 구조 + 빈셀 확인) ← 권장
$HANDLER read-table "양식.hwpx" -t 1
$HANDLER read-table "양식.hwpx" -t 3 --json  # JSON 형식
```

**template_map 자동 생성 (새 양식일 때):**
```bash
# 양식 분석 → template_map 초안 자동 생성
$PYTHON scripts/auto_template_map.py "양식.hwpx" -o template_map_draft.json

# kordoc parse_form 결과가 있으면 병합하여 정확도 향상
$PYTHON scripts/auto_template_map.py "양식.hwpx" -o template_map_draft.json \
  --kordoc-json form_fields.json
```
auto_template_map.py는 LABEL_KEYWORDS(성명, 주소, 사업명 등 40+개)로 라벨-값 쌍을 자동 감지하고,
테이블 용도(신청서/요약/인력/예산 등)를 자동 분류한다. 초안 생성 후 수동 검토·보정이 필요하다.

산출물: 테이블 인덱스, 행/열, 셀 매핑(빈셀 ✎ 표시), 병합 정보, 작성요령 테이블 위치

## Phase 3: 초안 작성

### 3-0. 광탈 패턴 사전 배제 (작성 전 체크)

초안 작성을 시작하기 전에 아래 광탈 패턴을 **절대 사용하지 않도록** 내부 규칙으로 설정한다.

**언어 광탈 패턴 (쓰면 감점):**
- "세계 최고 수준의 AI 기술" — 근거 없는 최상급 표현
- "국내 유일" / "독보적" — 검증 불가 주장
- "혁신적인" / "차세대" / "게임체인저" — 추상적 수식어
- "본 사업은 ~을 목적으로 하며" — AI 작성 흔적 1순위
- "이를 통해 ~의 효과를 기대할 수 있습니다" — AI 작성 흔적 2순위
- "~할 것으로 기대됩니다" — 막연한 미래 추정
- "4차 산업혁명" / "디지털 전환" — 남발된 유행어
- "글로벌 시장 진출" (6개월 안에 실현 불가한 목표)

**구조 광탈 패턴 (구성 시 주의):**
- 10페이지 중 7페이지 이상이 기술 설명 → "사업화 의지 부족"
- 정량적 목표가 0개이거나 너무 낮음 → "성의 없어 보임"
- 정량적 목표가 비현실적으로 높음 → "신뢰 불가"
- 예산 산출근거가 "1식" 단위 → 감점 확정
- 경쟁사 분석 없음 → 시장 이해도 부족
- 측정 불가능한 목표 (증빙 없이 달성 불가)

### 3-1. 평가기준 매칭 전략
각 평가항목(배점)에 회사의 강점/약점을 매칭하고, 섹션별 작성 전략을 수립한다.

### 3-2. 핵심 3문 답변 (작성 전 필수 확인)

협업 모드/자동 모드 모두에서 아래 3개 질문에 명확한 답을 먼저 정의한다. 이 답이 사업계획서 전체의 논리 축이 된다.

**질문 1: Why Now — 왜 지금인가?**
> 시장/기술/규제 환경에서 지금 이 시점이 특별한 이유는?
> (시장 성장률 데이터, 정책 변화, 경쟁 구도 변화 등 근거 포함)

**질문 2: Why Us — 왜 이 팀인가?**
> 경쟁사 대비 우리 팀이 이 문제를 더 잘 해결할 수 있는 구체적 근거는?
> (특허, 핵심인력 경력, 기존 고객사, 보유 데이터 등)

**질문 3: Why 지원 필요 — 지원 없이 왜 안 되는가?**
> 이 자금이 없으면 사업이 어떻게 지연/불가해지는가?
> ※ 심사역이 가장 중요하게 보는 질문. "이 회사가 자력으로 할 수 있으면 지원 불필요"

아래 세 가지를 회사 자료(재무제표)에서 직접 계산하여 수치로 제시한다. **AI가 임의로 숫자를 만들지 않는다.**

**① 자력 불가 증명 (현금 잔고 vs. 개발 비용)**
```
현재 현금 잔고:        X원  (재무제표 유동자산 기준)
이 사업 개발 비용:     Y원  (공고 지원금액 기준)
자체 조달 가능 금액:   Z원  (현금 × 가용 비율 추정)
갭:                   Y - Z = 부족분  → "자력 불가" 근거
```

**② BEP 시뮬레이션 (지원금 있을 때 vs. 없을 때)**
```
지원금 있을 경우:  M개월 후 손익분기점 도달 (Unit Economics 기반)
지원금 없을 경우:  개발 지연 N개월 → BEP N+개월 후로 밀림
```

**③ 런웨이 계산 (지원 없으면 언제 자금 소진)**
```
월 고정비: X원  (인건비 + 운영비)
현금 잔고: Y원
런웨이:    Y ÷ X = N개월 → 지원 없으면 N개월 후 자금 소진
```

재무제표 데이터가 없으면 사용자에게 정확한 수치를 요청한다. 추정치를 사용할 경우 반드시 "[추정]" 표시.

### 3-3-1. 본문 작성 편집 원칙

#### (a) 키워드 강조 (볼드+밑줄)
심사위원이 빠르게 핵심을 파악할 수 있도록, **중요 키워드에 볼드+밑줄을 적용**한다.

강조 대상 키워드:
- 핵심 수치: 매출액, 성장률, 투자금액, 사용자 수, 목표치
- Why Now 근거: 규제명(EU CSRD, SDGs 2030), 시장 규모, 타이밍
- Why Us 근거: 경쟁 차별점, 핵심 기술, 고유 자산
- 사업 성과: 고객사 수, 수상 이력, 투자 유치
- 정량 목표: 매출 증가율, 고용 계획, 산출물

HWPX 구현 방법:
- header.xml에 볼드+밑줄 charPr 스타일이 없으면 새로 등록 (bold={}, underline type="BOTTOM" shape="SOLID" color="#000000")
- 강조할 텍스트를 별도 `<hp:run>`으로 분리하여 강조 charPrIDRef 적용
- 비강조 텍스트는 기본 스타일(검정 맑은고딕) charPrIDRef 유지

DOCX 구현 방법:
```python
# python-docx에서 볼드+밑줄 적용
run = paragraph.add_run("강조 키워드")
run.bold = True
run.underline = True
```

#### (c) 전문용어 및 약어 설명 필수 표기
심사위원이 해당 분야 비전문가일 수 있으므로, 전문용어와 약어는 첫 등장 시 괄호로 풀이한다.

**자동 처리 절차** (Phase 3 초안 작성 시 필수 수행):
1. 초안 완성 후, 본문에 사용된 모든 약어·전문용어 목록 추출
2. 첫 등장 위치에 괄호 설명 삽입
3. 영문 약어는 **영문 풀네임 + 한국어 설명** 병기
4. 두 번째 이후 등장 시에는 약어만 사용
5. 요약표(T3)에서 처음 나오는 약어는 요약표 내에서 설명, 본문에서 처음 나오는 약어는 본문 내에서 별도 설명

예시:

- SaaS(Software as a Service, 클라우드 기반 구독형 소프트웨어)
- RAG(Retrieval-Augmented Generation, 검색 증강 생성)
- MVP(Minimum Viable Product, 최소 기능 제품)

#### (d) 개조식 명사형 문체 (필수)

### 3-4. 섹션별 초안 작성

**초안은 Phase 1에서 분석한 양식 구조를 기준으로 작성한다.** 일반적인 항목 순서가 아니라 실제 양식의 테이블맵·섹션명·ㅇ/- 단락 위치를 그대로 따른다. Phase 4의 DOCX/HWPX 생성은 이 초안을 그대로 옮기는 것이므로, 초안 단계에서 양식과 구조가 맞아야 한다.

**작성 전 확인:**
- `_사업계획서_준비자료.md`의 양식 테이블맵을 열어 두고 작성
- 각 섹션의 실제 이름(양식에 인쇄된 그대로), 테이블 번호, ㅇ/- 단락 수를 기준으로 작성
- 양식에 없는 섹션은 추가하지 않는다

**작성 순서:**
1. **요약표 초안 (T3)** — 양식의 요약표 셀 구조 그대로 한 줄씩 키워드 채우기
   - 본문 작성 후 8단계에서 최종 다듬는다
2. **본문 서술 섹션** — 양식의 섹션 순서대로, 각 ㅇ/- 단락 수에 맞춰 작성
   - 배점이 높은 섹션부터 공을 들인다 (평가기준표 참조)
   - 사업 내용 섹션: Why Now + Why Us(경쟁사 분석 구조) + Why 지원 필요(BEP·런웨이 수치) + Unit Economics
   - 기대효과 섹션: 계량화 수치 필수, "기여할 것으로 기대됨" 금지
   - 지역성 섹션: 양식에 있으면 반드시 포함
3. **테이블 데이터** — 인력(T5~T8), 정량목표(T10), 추진일정(T14), 예산(T15) 등 양식 테이블별로 행 단위로 작성
4. **요약표 최종 다듬기 (T3)** — 본문 전체 반영 후 첫인상 최적화

**ㅇ/- 단락 수 맞추기:**
양식에 ㅇ가 3개면 초안도 ㅇ 3개. 양식에 - 가 2개면 초안도 - 2개. 단락 수를 늘리거나 줄이지 않는다. 공간이 부족하면 한 ㅇ 안에 핵심만 압축한다.

#### Why Us 경쟁사 분석 구조 (신뢰도 높은 형식):

경쟁사 분석은 아래 구조로 작성한다. **차별점만 나열하지 않고, 우리가 못하는 것을 먼저 인정한 뒤 그럼에도 우리가 더 나은 이유를 제시**해야 심사역 신뢰도가 높아진다.

**직접 경쟁사 3개** (우리와 유사한 솔루션)

| 경쟁사 | 가격 | 주요 기능 | 주요 타겟 | 약점 |
|-------|------|---------|---------|------|
| A사   | 월 X만원 | … | … | … |
| B사   | 월 X만원 | … | … | … |
| C사   | 월 X만원 | … | … | … |

**간접 경쟁사 2개** (다른 방식으로 같은 문제를 해결)

| 경쟁사 | 해결 방식 | 한계 |
|-------|---------|------|
| D사   | … | … |
| E사   | … | … |

**대체재** (아무것도 안 쓰는 경우 / 엑셀·수기)
- 현재 시장의 X%는 대체재 사용 중 → 우리가 그들을 전환시킬 근거

**우리의 한계 인정 + 그럼에도 더 나은 이유:**
- 우리가 못하는 것: (예: "엔터프라이즈 레퍼런스 부족")
- 그럼에도 우리가 더 나은 이유: (예: "SMB 특화 가격·UI, 3개월 내 도입 가능")

#### 지역성 섹션 필수 포함 내용:
- 기업 소재지 및 지역 기반 운영 현황
- 지역 내 고객사/파트너 확보 계획
- 지역 일자리 창출 기여 (채용 계획 연계)
- 지역 산업 생태계(지역 특화 산업, 클러스터 등)와의 연계성
- 주관기관/지자체와의 시너지 포인트

#### 요약표(T3) 첫인상 최적화:
심사역은 요약표를 30초 안에 읽고 나머지를 볼지 결정한다. 각 셀의 목적:
| 셀 | 심사역에게 남겨야 할 인상 |
|---|---|
| 회사소개 | "이 회사는 실체가 있고 성과가 있다" |
| 주요사업 | "이미 시장 검증이 됐다" |
| 수상·선정이력 | "공신력 있는 기관이 인정했다" |
| 사업명 | "구체적이고 실현 가능해 보인다" |
| 사업배경·목적 | "왜 지금 필요한지 납득된다" |
| 주요수행내용 | "6개월 안에 실제로 할 수 있다" |
| 기대성과 | "측정 가능하고 현실적이다" |
| 정량적목표 | "달성 가능하고 증빙 가능하다" |
| 총사업비 | "규모가 적절하고 낭비가 없다" |

### 3-5. 팩트체크 & 리서치 보강

초안 작성 후, 아래를 자동으로 수행한다:

#### (a) 팩트체크
- 재무 수치가 제출 자료(재무제표)와 일치하는지 확인
- 매출 증빙 금액과 기재 금액이 일치하는지 확인
- 설립일, 주소 등 기본정보가 사업자등록증과 일치하는지 확인
- 기재한 수상/선정 이력이 참고자료에서 확인되는지 체크

#### (b) 시장 리서치 (WebSearch/WebFetch 사용)

**🚫 핵심 원칙: 검증 불가 데이터는 사실처럼 쓰지 않는다.**

##### 단계 1: 검색
- 타겟 시장 규모 데이터 검색 (예: "ESG 시장 규모 2026", "임팩트 측정 시장 성장률")
- 경쟁 서비스 현황 검색 (Why Us 근거 강화)
- 관련 정책/트렌드 뉴스 검색 (Why Now 근거 강화)

##### 단계 2: 검증
모든 시장 데이터(시장 규모, 성장률, 점유율, 통계 수치)는 아래 기준을 충족해야 한다:
- **출처 3요소 필수**: 보고서명(또는 기관명) + 발행연도 + URL(또는 접근경로)
  - 예: "한국임팩트투자네트워크, 2025 임팩트금융 현황 보고서, https://example.com/report"
- WebSearch로 찾은 수치는 **WebFetch로 원문 페이지를 열어 수치가 실제 존재하는지 재확인**
- 출처 3요소를 채울 수 없는 데이터는 **절대 사실로 기재하지 않는다**

##### 단계 3: 분류 후 기재
검증 결과에 따라 세 등급으로 분류한다:

| 등급 | 조건 | 기재 방식 |
|---|---|---|
| ✅ 검증됨 | 출처 3요소 확보 + 원문 확인 완료 | 수치 + 출처(기관명, 연도, URL) 본문 인라인 명기 |
| ⚠️ 추정치 | 검증된 상위 데이터에서 산출한 추정 | "~로 추정됨" + 산출 근거 + 원본 데이터 출처 URL |
| ❓ 확인 필요 | 출처 확보 실패 또는 원문 불일치 | "[확인 필요]" 태그 + 사용자에게 보완 요청 |

**외부 자료 출처 URL 필수 명기 규칙**:
- 시장 규모, 성장률, 통계, 정책 자료, 기술 동향 등 **외부 자료 인용 시 반드시 출처 URL을 본문에 직접 명기**
- 출처 형식: `(출처: 기관명, 보고서명, 발행연도, URL)` — URL 확보 불가 시 최소 `(출처: 기관명, 발행연도)`
- 각주가 아니라 **해당 수치/통계 바로 뒤에 괄호로 인라인 표기**
- 동일 출처 반복 인용 시 첫 인용만 URL 포함, 이후 `(출처: 상동)` 허용
- 예시:
  - 국내 AI 시장 약 45.3조원 규모 전망 (출처: 한국지능정보사회진흥원, 2025 AI 산업 실태조사, https://www.nia.or.kr/...)
  - 글로벌 SaaS 시장 CAGR 11.7% (출처: Grand View Research, 2024, https://www.grandviewresearch.com/...)

**금지 사항**:
- WebSearch 결과 없이 숫자를 생성하는 행위 (예: "약 X조 원 규모로 추정된다")
- 검색 결과의 숫자를 다른 맥락에 전용하는 행위
- 출처 없이 "~에 따르면", "업계에 따르면" 등의 표현 사용
- 출처 URL 없이 외부 통계/시장 데이터를 기재하는 행위

**⚠️ 시장 데이터 적합성 검증 (필수)**:
- 인용하는 시장 데이터가 신청 제품/서비스의 **직접 타겟 시장**인지 반드시 검증
- 심사위원이 "이 시장 데이터가 왜 이 제품과 관련 있는가?"라고 물었을 때 한 문장으로 답할 수 있어야 함
- 직접 시장 데이터가 없으면 상위 시장 데이터 + 연결 논리를 명시적으로 서술
- 예시:
  - (X) ESG 보고 소프트웨어 시장 → 임팩트 측정 서비스에 간접 관련 (시장 카테고리 불일치)
  - (O) IMM(임팩트 측정·관리) 시장 → 임팩트 측정 서비스에 직접 관련

#### (c) 논리 보강
- Why Now / Why Us / Why 지원 필요 세 논리가 모두 근거를 갖췄는지 점검
- 숫자 근거가 없는 주장에 데이터 추가
- "~할 것이다" 같은 막연한 표현을 구체적 수치로 보강
- 평가항목과 매칭되지 않는 빈 영역 식별 → 보완
- **광탈 패턴 최종 점검** — Phase 3-0의 목록과 대조

산출물: 초안 마크다운 + 팩트체크 결과 + 리서치 근거

### 3-6. 심사위원 관점 초안 검토 (Phase 3.5, 필수)

초안 작성 완료 직후, 휴먼라이징 전에 반드시 수행한다.
AI가 심사위원 역할을 맡아 초안을 검토하고, 약점을 개선한 뒤에만 다음 단계로 넘어간다.

> **왜 휴먼라이징보다 먼저?** 심사위원 검토에서 내용을 보강/수정하면 새로운 AI 표현이 추가될 수 있다. 내용 확정 후에 휴먼라이징을 해야 AI 흔적이 남지 않는다.

> **TIPS의 경우**: TIPS 전용 워크플로우 섹션의 "TIPS 심사위원 관점 초안 검토"가 이 단계를 대체한다. (4대항목 채점 + 킬러 질문 10개 + 약점 개선 + 재채점)

#### 검토 프로세스

```
초안 완료 → [1단계] 평가기준 채점 → [2단계] 핵심 질문 → [3단계] 개선 → [4단계] 재채점 → Phase 3.6 휴먼라이징
             항목당 100점            답변 불가 시          약점 보강        전 항목 60+ 확인
             60점 미만 있으면         개선 대상 식별
             다음 단계 진행 불가
```

#### 1단계: 평가기준별 모의 채점

Phase 1에서 추출한 **공고문의 평가기준**에 맞춰 채점한다. 심사위원 2인(현장 전문가, 심사 경력 심사역)의 관점을 시뮬레이션한다.

**채점 기준:**

| 점수 | 의미 |
|------|------|
| 90-100 | 탁월 — 이 항목만으로 선정 근거가 됨 |
| 70-89 | 우수 — 설득력 있으나 일부 보강 필요 |
| 60-69 | 보통 — 통과하지만 경쟁에서 밀릴 수 있음 |
| 40-59 | 미흡 — 이 항목에서 탈락 위험 |
| 0-39 | 치명적 — 이 항목이 전체 탈락을 결정 |

**공고별 평가기준 자동 매칭:**

Phase 1의 `_사업계획서_준비자료.md`에서 평가기준표를 읽어 항목별로 채점한다.
평가기준이 없는 공고의 경우 아래 범용 기준을 사용:

| 범용 항목 | 배점 | 핵심 질문 |
|----------|------|---------|
| 사업 이해도 | 20 | 공고 취지에 맞는 사업인가? 문제 정의가 구체적인가? |
| 기술/서비스 경쟁력 | 25 | 기술 차별점이 있는가? 경쟁사 대비 우위가 증명되는가? |
| 사업화 가능성 | 25 | BM이 명확한가? 매출 근거가 있는가? 실현 가능한 일정인가? |
| 수행 역량 | 15 | 팀 경력이 사업과 직결되는가? 파트너가 충분한가? |
| 기대효과 | 15 | 정량적 목표가 측정 가능한가? 사업비 대비 효과가 합리적인가? |

**채점 결과 출력:**

```
┌────────────────────────────────────────────────────┐
│ 심사위원 관점 모의 채점 결과                         │
├────────────────┬──────┬──────┬──────────────────────┤
│ 평가항목 (배점) │ 심사1 │ 심사2 │ 평균   판정          │
├────────────────┼──────┼──────┼──────────────────────┤
│ 사업이해도 (20) │  16  │  14  │  75   ✅ 통과        │
│ 기술경쟁력 (25) │  17  │  15  │  64   ⚠️ 경계        │
│ 사업화가능성(25)│  13  │  12  │  50   ❌ 미달 → 보강  │
│ 수행역량   (15) │  12  │  11  │  77   ✅ 통과        │
│ 기대효과   (15) │  10  │   9  │  63   ⚠️ 경계        │
├────────────────┼──────┼──────┼──────────────────────┤
│ 합계      (100)│  68  │  61  │  65   ⚠️ 통과이나 약점│
└────────────────┴──────┴──────┴──────────────────────┘

[약점 요약]
- 사업화가능성: 매출 산정 근거 부족, BM 설명이 추상적
- 기대효과: 정량 목표의 측정방법 미흡
```

**60점 미만 항목이 1개라도 있으면 다음 단계로 진행하지 않는다.**

#### 2단계: 핵심 질문 생성

심사위원이 실제 평가에서 물을 가능성이 높은 질문을 생성한다. 공고문의 평가기준에서 파생하되, 아래 범용 질문을 항상 포함:

**범용 핵심 질문 (모든 사업에 공통):**

1. "이 사업비 규모가 적정한 근거가 뭡니까? 절반으로 줄이면 안 됩니까?"
2. "경쟁사 대비 핵심 차별점이 뭡니까? 한 문장으로 말씀해 주세요."
3. "정량적 목표 달성을 어떻게 증명합니까? 증빙 서류가 뭡니까?"
4. "이 사업이 지원 없이 자체적으로 안 되는 이유가 뭡니까?"
5. "6개월 뒤 중간평가에서 보여줄 수 있는 산출물이 뭡니까?"

**공고 유형별 추가 질문:**

| 유형 | 추가 질문 |
|------|---------|
| 지역 R&D/사업화 | "이 사업을 이 지역에서 하는 이유가 뭡니까?", "지역 고용 효과는?" |
| 예비창업패키지 | "창업 경험이 없는데 실행 역량의 근거는?", "시장 검증은 했습니까?" |
| 창업도약패키지 | "기존 매출의 성장 정체 원인은?", "이 사업이 스케일업에 어떻게 기여합니까?" |
| 수출바우처/글로벌 | "수출 실적이 있습니까?", "타겟 국가의 규제/인증은 파악했습니까?" |

**답변 가능성 판정:**

| 판정 | 의미 | 조치 |
|------|------|------|
| ✅ 답변 충분 | 초안에 근거가 있고 논리적 | 유지 |
| ⚠️ 답변 약함 | 근거는 있으나 설득력 부족 | 보강 |
| ❌ 답변 불가 | 초안에 관련 내용 없음 | 필수 추가 |

#### 3단계: 약점 개선

1단계에서 60점 미만인 항목, 2단계에서 ❌/⚠️ 판정된 질문을 대상으로 초안을 개선한다.

**개선 우선순위:**
1. **❌ 답변 불가** → 해당 섹션에 내용 추가
2. **60점 미만 항목** → 해당 섹션 전면 보강
3. **⚠️ 답변 약함** → 근거 데이터 보충, 논리 강화

**개선 규칙:**
- 수치/데이터를 추가할 때 출처를 반드시 병기
- 개선 시 다른 섹션의 분량이나 논리를 훼손하지 않음
- 개선한 부분에 `[보강]` 마커를 달아 추적 가능하게 함
- 회사 자료에 없는 데이터를 AI가 지어내지 않음 — 자료 부족 시 사용자에게 요청

#### 4단계: 재채점 및 통과 확인

개선 후 1단계 채점을 다시 수행한다.

**통과 기준:**
- 평가항목 **전부 60점 이상**
- 핵심 질문 중 **❌ 답변 불가 0개**
- 광탈 패턴 **0개 검출**

**통과 실패 시:**
- 3단계→4단계를 **최대 2회** 반복
- 3회 반복해도 60점 미만 항목이 남으면 사용자에게 보고:
  - 어떤 항목이 약한지, 현재 점수와 목표 점수
  - AI가 보강하지 못하는 이유 (회사 자료 부족, 실적 부재 등)
  - 사용자가 추가로 제공해야 할 정보/자료 목록

**통과 시:**
- 채점 결과표를 `_심사위원_검토결과.md`에 저장
- Phase 3.6(휴먼라이징)로 진행

### 3-6-1. 휴먼라이징 (Phase 3.6, 필수)

심사위원 검토 통과 후, 시각 자료 생성 전에 반드시 수행한다.

> **이유**: 심사역은 하루 30개 이상의 계획서를 본다. AI 작성 문체는 3초 만에 식별되어 자동 감점된다. 휴먼라이징은 선택이 아니라 합격의 전제 조건이다.

수행 내용:

- 섹션별 AI 점수 확인 (목표: 전 섹션 30점 이하)
- 치환 후 문맥 자연스러움 재확인

### 3-7. 시각 자료 생성 (필수 — HWPX 작업 전에 완료)

**초안 완성 후, HWPX 채우기 전에 반드시 생성한다.** HWPX의 insert_image.py는 이 단계에서 만든 이미지 파일을 그대로 사용한다. 이 단계를 건너뛰면 HWPX에 이미지가 없는 채로 완성된다.

**생성할 이미지 목록을 초안에서 추출한다:**
- 초안에 "시장규모 차트", "매출 추이", "서비스 아키텍처", "경쟁사 비교" 등이 언급된 위치 확인
- 각 이미지에 파일명 지정 (예: `_시각자료/market_size.png`, `_시각자료/arch.png`)
- 이미지가 없으면 HWPX 삽입 단계에서 건너뜀 — **목록을 만들고 전부 생성 완료한 뒤 Phase 4로 진행**

**생성 전략 (혼합 방식 권장)**:
- **데이터 차트 + 인포그래픽** (시장규모, 매출추이, KPI 카드 등) → **matplotlib** 사용
  - 이유: 한국어 텍스트 정확, 수치 정밀 제어 가능
  - 폰트 자동 탐색 우선순위: Pretendard → NotoSansKR → AppleSDGothicNeo → Malgun Gothic → NanumGothic → AppleGothic
  - **브랜드 색상 자동 추출**: 로고·소개서·IR 이미지가 있으면 자동으로 색상 추출 + 가독성 보정 적용
  - **불필요 주석 금지**: 데이터와 출처만 표시, '본 사업 시점' 화살표 등 자의적 요소 추가 금지
- **개념도/아키텍처/로드맵** → **나노 바나나 프로(Gemini) 영문**으로 생성
  - 이유: Gemini 한국어 렌더링 불안정 → 텍스트 깨짐
  - 영문으로 생성해 HWPX에 삽입

```bash
PYTHON=~/.claude/skills/k-proposal/.venv/bin/python3
VGEN="$PYTHON ~/.claude/skills/k-proposal/visual_gen.py"

# ── Matplotlib 차트 / 인포그래픽 ──────────────────────────────────
# 단일 생성
$VGEN chart config.json -o _시각자료/market_size.png

# 변형 3개 생성 (팔레트별: 브랜드추출/블루/모노크롬)
$VGEN chart config.json -o _시각자료/market_size.png --count 3

# 브랜드 이미지 지정해서 색상 추출
$VGEN chart config.json -o _시각자료/market_size.png --brand-image 로고.png --count 3

# 현재 폴더에서 로고/소개서 이미지 자동 탐색해서 브랜드 색상 적용
$VGEN chart config.json -o _시각자료/market_size.png --auto-brand --count 3

# ── Gemini (아키텍처, 서비스 소개, 개념도) ──────────────────────────
$VGEN gemini "Service architecture diagram: mobile app → API server → AI engine → database" \
  -o _시각자료/arch.png --style diagram

# 변형 3개 생성
$VGEN gemini "..." -o _시각자료/arch.png --style diagram --count 3

# ── Mermaid (프로세스, 워크플로우) ──────────────────────────────────
$VGEN mermaid "graph TD; A[접수] --> B[심사] --> C[선정]" -o _시각자료/flow.png
```

**지원 차트 타입** (`config.json`의 `"type"` 값):

| 타입 | 설명 | 적합한 용도 |
|------|------|------------|

3. 기업정보 입력 (T1)
4. 인력/파트너/보유기술 입력 (T5~T8, **add-rows --template-row** 포함)

### HWPX 채우기 워크플로우 (검증된 4단계)

⚠️ remove-guides, insert-text는 파일 손상 위험이 높다. 대신 아래 워크플로우를 사용한다.

#### Step 2: 개조식 단락 채우기 (lxml 직접 조작)
hwpx_handler의 remove-guides/insert-text 대신 lxml로 직접 처리한다.
- 앞 단락의 고유 텍스트(예: "주력 사업 현황")를 키로 사용하여 정확한 위치 특정
- `<hp:t>` 텍스트를 직접 교체하거나, `<hp:t>`가 없으면 `<hp:run>` 안에 생성
- **인덱스 기반이 아닌 텍스트 패턴 매칭** 사용 (fill 후 인덱스 변경 위험 회피)

**SECTIONS 자동 생성 절차 (빈 배열로 두지 않는다):**

HWPX 양식의 개조식 단락 위치를 먼저 파악한 뒤, DOCX의 서술 내용을 그대로 옮긴다.

```bash
# 1) 양식에서 ㅇ/◦/- 로 시작하는 단락의 앞 헤더 텍스트를 추출
$HANDLER analyze "양식.hwpx" -t 0 -a | grep -E "^[IVⅠ-Ⅵ]|주력|사업|현황|배경|목적|기대"
```

추출한 헤더 텍스트와 DOCX에 작성한 내용을 매핑하여 SECTIONS를 완성한다:

```python

SECTIONS = [
    ('사업 배경 및 필요성', [        # HWPX 양식의 헤더 단락 텍스트

        ("   ㅇ [시급성 근거]",  "     - [정책/규제 변화]"),
    ]),
    ('주요 사업 내용', [
        ("   ㅇ [핵심 기능 설명]",   "     - [구체적 구현 방식]"),
        ("   ㅇ [차별점 — Why Us]",  "     - [경쟁사 대비 수치]"),
    ]),
    ('기대 효과', [
        ("   ㅇ [정량 목표 1]",  "     - [달성 시점 + 증빙 방법]"),
        ("   ㅇ [정량 목표 2]",  "     - [달성 시점 + 증빙 방법]"),

```

**체크**: SECTIONS를 완성한 뒤, 4-A-0 대조표의 "lxml Step 2" 행을 모두 ✅로 표시한다. 비어 있는 행이 있으면 SECTIONS에 추가하거나 fill로 처리한다.

#### Step 3: 후처리 (필수!)
```python
# 원본과 수정본의 모든 <hp:p> 텍스트를 비교
for i, (mod_p, mod_text) in enumerate(mod_texts):
    if mod_text != orig_text and mod_text.strip():
        # 1) linesegarray → 빈 태그 (글자 겹침 방지)
        for lsa in mod_p.findall(f'{HP}linesegarray'):
            mod_p.remove(lsa)
        etree.SubElement(mod_p, f'{HP}linesegarray')
        # 2) charPrIDRef → 79 (검정 맑은고딕, 파란색 가이드 스타일 제거)
        for run in mod_p.iter(f'{HP}run'):
            run.set('charPrIDRef', '79')
        # 3) 키워드 볼드+밑줄 (run 분할 + 강조 charPrIDRef 적용)
```

**볼드+밑줄 구현 방법:**
1. header.xml에서 기존 검정 맑은고딕 charPr(예: ID=79)를 복사
2. `<bold/>` + `<underline type="BOTTOM" shape="SOLID" color="#000000"/>` 추가
3. 새 charPr ID로 등록 (charProperties의 itemCnt 업데이트)
4. 강조할 키워드가 포함된 run을 분할: 일반 텍스트(charPr=79) + 강조 텍스트(charPr=새ID)

with zipfile.ZipFile("제출용.hwpx") as zf:
    root = etree.fromstring(zf.read('Contents/section0.xml'))

    t.text for t in root.iter(f'{HP}t') if t.text
)

checks = [

    "기대효과 수치",          # 예: "매출 X억", "고용 N명"
]
for kw in checks:
    status = "✅" if kw in all_text else "❌ 누락"

### 정량적 목표 작성 규칙 (T10)

## HWPX 파일 구조

> 상세 구조, 네임스페이스 매핑, 단위 변환, XML 요소 참조는 → **`references/xml-internals.md`** 참조

HWPX는 ZIP + XML 구조. 핵심 파일:
- `Contents/section0.xml` — 본문, 표, 이미지, 텍스트
- `Contents/header.xml` — 스타일(charPr, paraPr), 폰트
- `Contents/content.hpf` — OPF manifest (이미지 등록)
- `BinData/` — 이미지 파일
- `mimetype` — 반드시 첫 엔트리, ZIP_STORED

핵심 XML 경로:
- **테이블**: `hp:tbl` > `hp:tr` > `hp:tc` > `hp:subList` > `hp:p` > `hp:run` > `hp:t`
- **이미지**: `hp:p` > `hp:run` > `hp:pic` > `hc:img[binaryItemIDRef]`
- **텍스트**: `hp:p` > `hp:run[charPrIDRef]` > `hp:t`

## 주의사항

### 공통 주의사항
1. **분량 제한**: 공고문에 명시된 페이지 이내로 작성
2. **팩트체크 필수**: 재무 수치, 이력, 날짜는 반드시 원본 자료와 대조
3. **AI 흔적 제거 타이밍**: 초안 완성 직후, 문서 생성 전에 반드시 수행
4. **증빙 가능성**: 정량 목표는 반드시 증빙 가능한 형태로 설정
5. **예산 산출근거**: "1식" 단위 금지, 공수 기반으로만 작성

### DOCX 주의사항 (Phase 4-opt 실행 시에만 해당)
6. **양식 구조 파악 필수**: HWPX analyze로 원본 양식의 섹션 구조(Ⅰ~Ⅵ)와 개조식 서술 영역(ㅇ / - / ◦)을 먼저 파악한 뒤 DOCX 생성
7. **시각 자료 삽입**: `_시각자료/` 폴더의 이미지를 doc.add_picture()로 삽입 (Phase 3-7에서 이미 생성된 파일 사용)
8. **폰트**: macOS에서는 AppleGothic, Windows에서는 맑은 고딕 사용. DOCX 스타일에서 font.name 지정
9. **제출 변환**: DOCX → HWP가 필요하면 한글에서 DOCX를 열어 HWP로 저장. DOCX → PDF는 워드에서 변환

### HWPX 주의사항
10. **스타일 보존**: 셀 텍스트 쓸 때 기존 `charPrIDRef` 유지 (preserve_style=true)
11. **셀 병합 주의**: read-table의 `[NxM]` 표시로 병합 셀 확인 후 올바른 col 인덱스 사용
12. **fill 전 반드시 validate**: `fill --validate`로 좌표 오류를 사전 확인 (재시도 방지)
13. **행 추가 시 template-row**: 합계행이 아닌 데이터 행을 복제하도록 `--template-row` 지정 권장

### HWPX 편집 시 알려진 한계 (⚠️ 중요)

> 상세 원인·해결책 매트릭스는 → **`references/xml-internals.md`** "알려진 한계 및 해결책" 참조

주요 문제와 대응:

| 문제 | 대응 |
|------|------|
| 네임스페이스 프리픽스 깨짐 (ns0:/ns1:) | `fix_namespaces.py` 후처리 (필수) |
| 글자 겹침 | diff 기반 linesegarray 빈 태그 교체 |
| 파란색 글자 | charPrIDRef → 79(검정) 변경 |
| XML 선언 따옴표 | 큰따옴표 정규표현식 치환 |
| ZIP 메타데이터 변경 | 원본 ZIP 정보 복원 |
| 네임스페이스 선언 누락 (16개 필수) | register_namespace + fix_namespaces |

**대응 전략**:
- HWPX fill 후 반드시 **postprocess_hwpx.py 실행** (diff → lineseg + charPr + 볼드 + 네임스페이스 정규화)
- **remove-guides, insert-text는 파일 손상 위험** → 개조식 단락은 lxml 직접 조작(텍스트 패턴 매칭)으로 채움
- HWPX가 열리지 않으면 사용자에게 알리고 Phase 4-opt(DOCX 생성) 진행 여부를 확인
- HWPX 편집 후에는 반드시 한글에서 열리는지 확인한 뒤 다음 단계 진행
- **손상 ZIP 자동 복구**: hwpx_handler.py가 자동으로 Local File Header 스캔 방식으로 복구 시도

## Markdown → HWPX 하이브리드 워크플로우 (대안)

기존 워크플로우(lxml 직접 조작)가 복잡한 경우, 마크다운 초안을 HWPX로 변환하는 하이브리드 접근도 가능하다.

**사용 시나리오:**
- 자유 서술 섹션이 많고 테이블이 적은 양식
- 새 양식의 빠른 프로토타이핑
- kordoc MCP가 설치된 환경

**하이브리드 워크플로우:**
```
Phase 3 초안(md) → kordoc markdownToHwpx → 기본 HWPX 골격 생성
                                              ↓
                        원본 양식 + 기본 골격 → 수동 병합 (테이블 데이터는 기존 fill 사용)
```

**주의사항:**
- kordoc의 markdownToHwpx는 MVP 수준으로 기본 단락·제목·테이블만 지원
- 원본 양식의 정확한 레이아웃 재현이 필요하면 기존 lxml 워크플로우 권장
- 하이브리드는 서술 영역 초안 → HWPX 변환 후 원본 양식에 병합하는 방식

### TIPS 전용 핵심 3문

Phase 3 초안 작성 전에 아래 3개 질문에 명확한 답을 먼저 정의한다.

**Why Now — 왜 지금 이 R&D가 필요한가?**
> 시장/규제/기술 환경에서 지금 이 시점이 특별한 이유.
> (시장 성장률, 정책 변화, 기술 전환점 등 근거 포함)

**Why Us — 왜 이 팀인가?**
> 특허/논문/핵심인력 경력으로 경쟁사 대비 우위 증명.
> (IP 목록, CTO/CSO 경력, 기존 PoC/매출 실적)

**Why 정부 R&D — 투자만으로 왜 안 되는가?**
> 선투자 2억 + R&D 8억이 왜 모두 필요한지, 자력 불가 증명.
> - 현재 현금 잔고 vs. R&D 비용 갭
> - BEP 시뮬레이션 (R&D 있을 때 vs. 없을 때)
> - 런웨이 계산 (R&D 없으면 N개월 후 자금 소진)

### TIPS 연구개발계획서 본문 구성 가이드

#### 섹션별 분량 비율 (8억 규모 기준)

8억 R&D에 걸맞는 밸런스. 기술만 가득하면 "학술 논문", 사업화만 가득하면 "R&D 의지 부족".

- **기술 영역 40%** (필요성 + 목표/내용 + 추진전략 + 연구현황)
  - 시스템 아키텍처, TRL, 성능지표, 기술적 난제와 해결방안
  - 8억 규모 = 연구원 3-5명 × 2년 풀타임 → 논문 2편+, 특허 3건+ 수준의 R&D 산출물 기대
- **사업화 영역 30%** (사업화전략 + 기대효과 + 후속투자시나리오)
  - BM, 경쟁사 비교, 매출산정근거, 글로벌, 투자라운드 로드맵
- **팀/실적 영역 30%** (기업소개 + 추진체계)
  - CTO/CSO 역할, 관련 경력, PoC/매출 실적, 파트너십

기술 비중 50% 초과 또는 사업화 비중 15% 미만 시 경고 출력.

#### 합격률 높은 본문 작성 순서

1. **연구개발과제의 필요성** (T26): Top-Down(거시 트렌드 → 구체 문제), 수치+그래프, 시장의 빈틈
2. **해결방안** (T27): 핵심 솔루션 요약 → 기술 차별점 → 적용 시나리오
3. **연구개발 목표 및 내용** (T28): 시스템 아키텍처 다이어그램, 알고리즘 플로우차트, TRL 현재→목표
4. **연구개발 현황** (T34): 보유 IP, MVP/프로토타입, 확보 데이터, 선행연구
5. **사업화 전략** (T39, T41): 경쟁사 비교표, BM, 초기 고객 확보, 매출산정근거(TAM/SAM/SOM)
6. **후속 투자 시나리오** (T41 내): TIPS R&D → Post-TIPS → 시리즈A → 스케일업 성장 사다리
7. **기업소개** (T52): 대표/CTO/CSO 역할분담, 문제와 직결되는 경력, 팀 시너지

### TIPS 성능지표 작성 규칙 (최종평가 역설계)

성능지표(T29)는 선정 평가의 핵심이자 2년 후 최종평가의 판정 기준.

**정량 수치 규칙:**
- 측정 가능한 정량 수치만 ("높은 정확도" X → "예측 정확도 80% 이상" O)
- 세계최고 + 국내 수준 벤치마크 기재 (논문/특허/제품 스펙 근거)
- 비중(%) 합계 100%

**평가방법 신뢰도 순서 (T31):**
1. 공인시험기관 성적서 (KTL, KCL, KTC)
2. 고객사/수요기관 PoC 확인서
3. 자체 평가 (환경/도구/절차 구체 명시 필수)

**최종평가 역설계 (선정 + 성공판정 동시 달성):**

| 세계최고 대비 구간 | 선정 가능성 | 최종평가 위험 | 판정 |
|-----------------|-----------|------------|------|
| 50% 미만 | 낮음 (야심 부족) | 안전 | 상향 필요 |
| 60-80% | 적정 | 안전 | **권장 구간** |
| 80-90% | 높음 (도전적) | 보통 | 충분한 근거 시 허용 |
| 90-100% | 매우 높음 | 높음 | 경고 |
| 100% 초과 | 비현실적 판단 | 매우 높음 | 금지 |

- 모든 지표에 달성 가능성 80% 이상을 내부 기준으로 설정
- 중간점검 마일스톤: 각 지표에 "1년차 60% 달성" 체크포인트 병기
- 최종평가: 우수(90%+), 보통(60-89%), 미흡(60% 미만) → 보통 이상 확보가 최소 목표

### TIPS 후속 투자 시나리오 가이드

사업화 전략 섹션(T41)에 투자 로드맵 필수 포함:

```
현재 ─── TIPS R&D(2년) ──── Post-TIPS ────── 시리즈A ────── 스케일업
선투자2억  R&D 8억(정부)    추가 7억(정부)   민간투자      매출기반성장
           기술 완성         사업화 가속       시장 확대      글로벌 진출
```

각 단계별 연동 목표:
- TIPS R&D 완료: TRL 6→8, 핵심 특허 3건, PoC 고객 5곳, 매출 X억
- Post-TIPS: 제품 출시, 매출 Y억, 후속 투자 Z억 유치
- 시리즈A: 시장 점유율 N%, 글로벌 진출 M개국, 기업가치 XX억

작성 규칙:
- TIPS R&D 성과가 다음 투자 라운드의 "증거"가 되도록 목표 설계
- 매출 예측은 TAM/SAM/SOM 기반 산출근거 명시
- 운영사의 follow-on 투자 의향이 있으면 반드시 언급
- "TIPS R&D 없으면 이 사다리를 탈 수 없다"는 논리와 Why 정부 R&D를 연결

### TIPS 예산 작성 규칙

정부출연금 최대 8억원 + 민간부담금(현금+현물).

**비목별 한도:**
- 인건비: 계상률 100% 이내, 2026년 단가 — 책임연구원 378만원/월, 연구원 290만원/월
- 위탁연구개발비: 직접비의 40% 이내
- 외부전문기술활용비: 직접비의 40% 이내
- 간접비: 직접비의 10% 이내 (영리기관)
- 연구수당: 인건비의 20% 이내

**산출근거 규칙:**
- "1식" 단위 감점. 반드시 공수(인원 × 기간 × 단가) 기반 작성
- 나쁜 예: "AI 엔진 개발 1식 = 50,000천원"
- 좋은 예: "AI 엔진 고도화 용역: 연구원 2인 × 6개월 × 월 2,901,312원 = 34,816천원"

**민간부담금 구성:**
- 현금 비율 60% 이상 권장 (사업 의지 바로미터)
- 운영사 선투자금이 민간부담금 현금에 포함
- 현물만으로 구성 시 "정부 돈만 타려는 것"으로 의심

### TIPS 심사위원 관점 초안 검토 (Phase 3 완료 후, HWPX 채우기 전 필수 수행)

초안 작성(Phase 3) 완료 직후, 휴먼라이징(Phase 3.6) 전에 반드시 수행한다.
AI가 TIPS 심사위원 역할을 맡아 초안을 검토하고, 약점을 개선한 뒤에만 Phase 4로 넘어간다.

#### 검토 프로세스

```
초안 완료 → [1단계] 4대항목 채점 → [2단계] 킬러 질문 → [3단계] 개선 → [4단계] 재채점 → Phase 4
               60점 미만 있으면        답변 불가 시           약점 보강        전 항목 60+ 확인
               Phase 4 진행 불가       개선 대상 식별
```

#### 1단계: 서류평가 4대항목 모의 채점

심사위원 3인(기술 전문가, 사업화 전문가, 투자자)의 관점을 시뮬레이션하여 각 항목을 채점한다.

**채점 기준:**

| 점수 | 의미 | 판단 기준 |
|------|------|---------|
| 90-100 | 탁월 | 이 항목만으로 선정 근거가 됨 |
| 70-89 | 우수 | 설득력 있으나 일부 보강 필요 |
| 60-69 | 보통 | 통과하지만 경쟁에서 밀릴 수 있음 |
| 40-59 | 미흡 | 이 항목에서 탈락 위험 |
| 0-39 | 치명적 | 이 항목이 전체 탈락을 결정 |

**항목별 채점 포인트:**

**(1) 문제인식 (T26 필요성)**
- 시장 문제가 수치로 증명되었는가? (데이터 출처 포함)
- Top-Down 구조가 논리적인가? (거시→미시 흐름)
- "왜 지금"의 답이 명확한가? (정책 변화, 기술 전환점)
- 기존 솔루션의 한계가 구체적인가?
- 감점: "불편하다" 수준의 추상적 문제 정의, 출처 없는 시장 데이터

**(2) 실현가능성 (T27-T29 해결방안+내용+성능지표)**
- 시스템 아키텍처 다이어그램이 있는가?
- 핵심 기술의 모듈 분해가 되어 있는가? (추상적 설명 vs 구체적 구현)
- TRL 현재→목표가 명시되었는가?
- 6개월 단위 로드맵이 현실적인가?
- 성능지표가 정량적이고 측정 가능한가?
- 세계최고 대비 목표치가 60-80% 권장 구간인가?
- 감점: "AI 기반 시스템 개발" 수준의 추상적 기술 서술, TRL 미기재

**(3) 성장전략 (T38-T41 경쟁사+시장+사업화)**
- BM이 명확한가? (누가 돈을 내는가)
- 경쟁사 비교가 기능/가격/성능 비교표로 되어 있는가?
- 매출 산정에 TAM/SAM/SOM 근거가 있는가?
- 후속 투자 시나리오(TIPS→Post-TIPS→시리즈A)가 있는가?
- 글로벌 진출 계획이 있는가?
- 감점: 경쟁사 분석 없음, 매출 근거 없는 숫자, 후속 계획 부재

**(4) 팀 구성 (T52 기업소개)**
- 2인 이상 팀인가?
- CTO/CSO 역할 분담이 명확한가?
- 팀원 경력이 해결하려는 문제와 직접 관련 있는가?
- 팀 시너지(기술+사업+마케팅)가 설명되어 있는가?
- 감점: 대표 1인 구성, 역할 미분리, 문제와 무관한 경력 나열

**채점 결과 출력 형식:**

```
┌─────────────────────────────────────────────────────┐
│ TIPS 서류평가 모의 채점 결과                          │
├──────────┬──────┬──────┬──────┬──────────────────────┤
│ 평가항목  │ 기술 │ 사업 │ 투자 │ 평균   판정          │
├──────────┼──────┼──────┼──────┼──────────────────────┤
│ 문제인식  │  75  │  70  │  72  │  72   ✅ 통과        │
│ 실현가능성│  65  │  60  │  55  │  60   ⚠️ 경계        │
│ 성장전략  │  50  │  65  │  70  │  62   ⚠️ 경계        │
│ 팀 구성  │  60  │  55  │  58  │  58   ❌ 미달 → 보강  │
├──────────┼──────┼──────┼──────┼──────────────────────┤
│ 종합     │      │      │      │  63   ⚠️ 통과이나 약점│
└──────────┴──────┴──────┴──────┴──────────────────────┘
```

**60점 미만 항목이 1개라도 있으면 Phase 4로 진행하지 않는다.**

#### 2단계: 킬러 질문 10개

심사위원이 실제 대면평가에서 물을 가능성이 높은 날카로운 질문 10개를 생성한다. 초안의 내용만으로 답변이 가능한지 테스트한다.

**필수 킬러 질문 (초안에서 반드시 답할 수 있어야 하는 것):**

1. "이 기술이 8억짜리 R&D인 이유가 뭡니까? 2억이면 안 됩니까?"
2. "세계 최고 수준 대비 XX%를 목표로 했는데, 스타트업 규모에서 가능한 근거는?"
3. "경쟁사 A가 이미 이걸 하고 있는데, 차이점이 뭡니까?"
4. "선투자 받은 운영사가 이 기술의 어떤 점에 투자한 겁니까?"
5. "2년 뒤 최종평가에서 성능지표 달성이 안 되면 어떻게 합니까?"
6. "CTO가 이 분야 경험이 X년인데, 핵심 기술적 난제를 풀 수 있다는 근거는?"
7. "매출 목표 XX억의 산출 근거가 정확히 뭡니까?"
8. "TIPS R&D 없이 자체 자금으로는 왜 안 됩니까? 런웨이가 얼마입니까?"
9. "글로벌 진출 계획이 있는데, 어느 나라부터, 어떤 채널로 가는 겁니까?"
10. "Post-TIPS, 시리즈A 계획이 있는데, 구체적 투자자 접촉이 있습니까?"

**답변 가능성 판정:**

| 판정 | 의미 | 조치 |
|------|------|------|
| ✅ 답변 충분 | 초안에 근거가 있고 논리적 | 유지 |
| ⚠️ 답변 약함 | 근거는 있으나 설득력 부족 | 보강 |
| ❌ 답변 불가 | 초안에 관련 내용 없음 | 필수 추가 |

#### 3단계: 약점 개선

1단계에서 60점 미만인 항목, 2단계에서 ❌/⚠️ 판정된 질문을 대상으로 초안을 개선한다.

**개선 우선순위:**
1. **❌ 답변 불가** 항목 → 해당 섹션에 내용 추가
2. **60점 미만** 항목 → 해당 섹션 전면 보강
3. **⚠️ 답변 약함** 항목 → 근거 데이터 보충, 논리 강화

**개선 규칙:**
- 개선 시 기존 분량 비율(기술 40%/사업화 30%/팀 30%)을 유지
- 한 섹션을 보강할 때 다른 섹션의 내용을 줄이지 않음
- 수치/데이터를 추가할 때 출처를 반드시 병기
- 개선한 부분에 `[보강]` 마커를 달아 추적 가능하게 함

#### 4단계: 재채점 및 통과 확인

개선 후 1단계 채점을 다시 수행한다.

**통과 기준:**
- 4대 항목 **전부 60점 이상**
- 킬러 질문 10개 중 **❌ 답변 불가 0개**
- 광탈 패턴 **0개 검출**

**통과 실패 시:**
- 3단계→4단계를 **최대 2회** 반복
- 3회 반복해도 60점 미만 항목이 남으면 사용자에게 보고:
  - 어떤 항목이 약한지
  - 왜 AI가 보강하지 못하는지 (회사 자료 부족, 실적 부재 등)
  - 사용자가 추가로 제공해야 할 정보/자료 목록

**통과 시:**
- 채점 결과표를 `_TIPS_심사_검토결과.md`에 저장
- Phase 4(HWPX 채우기)로 진행
